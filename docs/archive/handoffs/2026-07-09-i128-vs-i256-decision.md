# Handoff: i128 vs i256 decision for the LEZ weighted-pool `pow` kernel

## What this conversation was

A design discussion for the `rust-pow-package` repo (`weighted-math-core` crate) — a fixed-point weighted-pool `pow` kernel for a Balancer-style LBP running as a RISC Zero zkVM guest on the Logos Execution Zone (LEZ). See `CONTEXT.md` for the full design brief and `docs/adr/0001-open-decisions.md` for the open decisions. No code was written or changed this session — it was all analysis and a decision.

The active question: should the kernel use `i128`/`u128` (with localized widening) or `i256`/`u256` for its fixed-point arithmetic? Secondary topic touched but not deeply pursued: the `no_std` requirement for the guest.

## Decision reached (recommendation, user agreed "makes sense")

Use `i128`/`u128` plus the single localized `128x128 -> 256` outer multiply already specified in the brief. Do NOT promote the whole kernel to 256-bit. Keep i128 as the default unless the mpmath accuracy sweep proves it cannot hit the target error bound across the exponent range (0.0101 to 99); if it can't, widen selectively at the specific step that needs it, not everywhere.

Three independent reasons, all pointing the same way:
1. It is the Logos platform default (see evidence below).
2. It is cheaper in the RISC0 cost model (32-bit native word; each width doubling multiplies the division-heavy pow cost ~2-4x).
3. It is the smaller overflow-proof and audit surface (this is an audited RFP-016 artifact).

## Key evidence — Logos ecosystem defaults (the decisive part)

Researched the `logos-co` GitHub org and the LEE v0.3 spec to avoid assuming anything about chain cost. Findings:

- **Token amounts / on-chain money math are `u128` everywhere.** LEE v0.3 built-in program spec: `logos-co/logos-lips` at `docs/blockchain/raw/lez/lee-v0.3-specifications/appendices/builtin-programs.md` — Token Program balances, supply, mint/burn all `u128`.
- **The spec's own reference AMM is `u128` throughout.** Same file has a "Liquidity Pool Program" (constant-product AMM). State is `reserve_a/reserve_b/liquidity_pool_supply: u128`; swap math is plain `u128`: `amount_out = (reserve_out * amount_in) / (reserve_in + amount_in)`. Closest possible precedent — an on-chain AMM in Logos's own spec doing multiply-then-divide in u128, accepting a bounded overflow envelope rather than widening.
- **Overflow handled by checked/saturating arithmetic on u128, not widening.** `logos-co/lez-payment-streams` at `lez-payment-streams-core/src/vault.rs` uses `checked_add`/`checked_sub`/`saturating_sub` on u128.
- **SPEL framework codec: `i128` is a first-class primitive; no `u256`/`i256` exists.** `logos-co/spel` at `spel-framework-core/src/decode.rs` decodes `u8/u16/u32/u64/u128/i128` and nothing wider.
- **`U256`/`ruint` appear ONLY at the Ethereum boundary** (e.g. `logos-co/eth-lez-atomic-swaps` `src/eth/client.rs`, `logos-evm-wallet-backend-module`). Inside LEZ guests, 256-bit is a foreign EVM type, never a native choice.
- **No fixed-point math exists anywhere in the ecosystem yet** (`fixed_point` code search: 0 hits). This kernel is the first, so "stick to defaults" = the u128/i128 + checked-arithmetic + localized-widening pattern, since it's the only pattern that exists.

Existing LEZ program repos for reference: `logos-co/lez-framework`, `lez-multisig`, `eth-lez-atomic-swaps`, `logos-lez-rln`, `lez-payment-streams`, `spel`.

## Cost analysis (secondary, estimated not measured)

Traced Balancer v3 to count operations (commit `7861ea2785b96dd10681ff1b8dfe56b36cc202b6`):
- `LBPool.onSwap` -> `WeightedMath.computeOutGivenExactIn` = 2 multiplies + 2 divides + **1 `powUp`**. The pow is ~99% of the cost.
- One `pow(x,y) = exp(y*ln x)` in `LogExpMath.sol` runs roughly a couple dozen divides and ~30 multiplies (range-reduction ladders + Taylor series: ln ~6 terms, exp ~12 terms). Divides dominate — and division is the most expensive RISC0 primitive per CONTEXT.md.
- Our kernel is lighter than generic Balancer pow: base<1 & exponent>0 deletes the large-argument exp branch, and range reduction is by ln2 via **shifts, not divides**.
- Rough zkVM estimate: pow ~single-digit-thousands of cycles at u128, ~3x that at u256; the i128-vs-i256 delta is a few thousand cycles per swap against a ~2^20 (~1.05M) cycle RISC0 segment — well under 1% of one proof. So width barely moves per-tx cost.
- EVM contrast (why AMMs use 256): EVM native word IS 256-bit, so wide is free there and narrow costs extra masking gas. On RISC0 (32-bit native word) the economics invert: wide is emulated and expensive. That's why "every AMM uses 256" does not transfer.

Corrections made mid-conversation (do not repeat the earlier overstatements): Balancer's math is NOT incomplete — the `expm1` handling in the brief is an optional precision refinement WE choose for sale-start catastrophic cancellation, not a gap in Balancer. The base-range difference is a simplification in our favor, not a problem. i256 was never "prohibited" — it's a cost/precedent tradeoff.
